## 1. 产品定位

### 1.1 一句话定义

> 一款由 5 个像素风外星人组成的 AI 原生桌面陪伴产品，每个角色拥有独立世界观与知识库，通过 RAG 实现角色化讲述、对话与小游戏交互。

### 1.2 区别于已有产品

|  | 传统桌宠（VPet/DesktopMate） | AI 陪伴（Replika/Character.AI） | **像素星球** |
| ---- | ---- | ---- | ---- |
| 视觉 | 2D/3D 写实或动漫 | 头像 + 文字 | **像素 + 多动画状态** |
| 角色 | 单角色 | 用户自建 | **5 个固定角色 + 完整世界观** |
| 交互 | 桌面行为为主 | 纯对话为主 | **桌面行为 + 对话 + 小游戏 + 故事** |
| AI | 基本无 | 通用 LLM | **角色化 RAG + 情绪状态机** |
| 入口 | 仅桌面端 | 仅 App | **网页星球门户 + 桌面端** |

### 1.3 价值主张

- 对用户：在屏幕上拥有 5 个有"宇宙背景"的伙伴，不只是个表情包，而是有持续叙事的角色。
- 对作者：验证"纯 AI 工具链单人完成创意产品 0→1"的可行性，沉淀 RAG 实战经验。

## 2. 目标用户

### 2.1 核心人群（P0）

- **数字原住民程序员/学生**，18-30 岁，长时间面对屏幕，喜欢像素风、独立游戏文化，愿意为屏幕陪伴付出时间但厌倦"打卡式社交"。
- 典型需求：写代码时屏幕边角有个伙伴 / 长时间专注后有人提醒喝水 / 偶尔切角色获得情绪刺激。

### 2.2 次级人群（P1）

- **创意工作者 / 内容创作者**：喜欢截图分享桌面，桌宠是社交资本。
- **Replika / 星野 重度用户**：寻找比纯文字更轻、比 VRoid 更不"压力"的陪伴产品。

### 2.3 不服务

- 寻找成人/恋爱向 AI 伴侣的用户
- 需要专业生产力工具的用户

## 3. 核心场景

### 场景一：第一次落地星球（首次使用）

> 用户打开网页/桌面端 → 看到 5 个外星人的星球门户 → 点击 SPA-001 进入像素族生物档案 → 在终端输入 hi → SPA-001 用自己的语气回应：「检测到友好光标，初次握手。」

**关键设计**：开箱即用，无需注册即可与 5 个角色对话。

### 场景二：编程专注陪伴（高频）

> 用户切到 ALN-003 Lumi → 桌面端持续显示 idle 状态 → 写代码超过 25 分钟 → Lumi 触发 yawning 动画并气泡："要不要起来走走？" → 编译成功 → Lumi 触发"编译祝福"动画。

### 场景三：故事时间（情感粘性）

> 用户晚上对 ALN-005 赤炎精灵输入「讲个故事」→ RAG 检索 Pyros-3 世界观片段 → LLM 用赤炎精灵的口吻生成 300-500 字故事流式输出 → 故事缓存供下次回看。

### 场景四：角色切换（探索性）

> 用户从 ALN-002 切到 ALN-004 → 网页配色从绿色变暖橙 → 终端 prompt 从 `aln-002` 变 `aln-004` → 角色记忆相互独立。

## 5. 核心能力详细设计：RAG 角色化对话

### 5.1 数据流

```
用户输入
  └─→ ① 当前角色识别（data-species）
       └─→ ② 角色专属向量库检索 top-3 知识片段
            └─→ ③ Prompt 拼装（角色 Bible + 检索片段 + 最近 6 轮对话 + 当前 PAD）
                 └─→ ④ LLM 流式调用（通过边缘函数代理）
                      └─→ ⑤ 流式输出到气泡 + 解析情绪变化 → 更新 PAD
                           └─→ ⑥ 写入对话历史 + 触发对应动画状态
```

### 5.2 知识库结构

每个角色独立 namespace，chunk 来源：
- 该角色的 Bible（`docs/character-bibles/<id>.md`）切片
- 该角色 20-30 句标志性台词
- 角色相关故事种子 3-5 条

总 chunk 量预估：5 角色 × 40 chunk ≈ 200 条，**单机内存向量库即可**，不需要重型向量库。

### 5.3 Prompt 模板

```
你是 {nameCN}（{species}），来自 {motherPlanet} 星 {sector} 扇区。

【性格】{personality}
【语言风格】{voiceStyle}
【你绝不会做的事】{taboos}
【当前情绪】PAD = ({P}, {A}, {D})

【与你相关的设定】
{retrieved_chunks}

【最近对话】
{history}

请用第一人称、保持你的语言风格，回复用户：
{user_input}

回复后请在末尾用 JSON 给出情绪变化，例如：[ΔPAD: {"P": 0.1, "A": 0.0, "D": 0.0}]
```

### 5.4 不做（边界）

- 不做长期记忆压缩（v1.0 范围内只保留最近 6 轮）
- 不做 function calling（先纯对话）
- 不做 TTS（v2.0 再做）
- 不做用户隐私加密（仅 localStorage，文档明示）

## 6. 非功能需求

| 维度 | 要求 |
| ---- | ---- |
| 性能 | 角色切换 < 200ms；对话首字响应 < 1.5s |
| 可用性 | 网页支持 Chrome 100+ / Safari 16+ |
| 桌面端 | macOS 12+ / Windows 11，安装包 < 30MB（Tauri 优势） |
| 隐私 | 默认本地存储；仅对话内容上行至 LLM API；明示告知 |
| API 成本 | 单用户单日 < ¥0.5，依赖小模型 + 短上下文 |

## 8. 数据指标定义（为后续运营预埋）

- **DAU / WAU**：访问网页或启动桌面端
- **角色偏好分布**：5 角色被选中比例（看哪个角色最受欢迎）
- **对话轮次**：人均/角色平均对话轮次
- **次日留存**：访问后第二天回访率
- **故事完播率**：故事开始后看完比例

## 9. 风险与对策

| 风险 | 对策 |
| ---- | ---- |
| LLM API key 泄漏 | 边缘函数代理 + 速率限制 |
| 角色"出戏"（说不符合人设的话） | Prompt 加"绝不会做的事"段落 + 关键词后处理过滤 |
| 内容生成成本失控 | 默认小模型（DeepSeek/Haiku），缓存常见对话 |
| 5 角色让用户决策疲劳 | 首页推荐"今日推荐角色"或基于时间段（夜晚推 ALN-002） |
